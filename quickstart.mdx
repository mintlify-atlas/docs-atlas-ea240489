---
title: "Quickstart: Build and deploy your first SQL project"
description: "Create a SQL project, add database objects, build a dacpac, and publish it to a SQL Server or Azure SQL database in under 10 minutes."
---

In this quickstart you will install SqlPackage and the SQL project templates, scaffold a new project, add tables and a stored procedure, build a `.dacpac`, and publish it to a database. By the end you will have a working SQL project ready to extend and deploy through CI/CD.

<Note>
  Microsoft.Build.Sql requires .NET 8 or higher. Verify your installation with `dotnet --version` before starting.
</Note>

<Steps>
  <Step title="Install SqlPackage">
    Install SqlPackage as a .NET global tool. It is available for Windows, macOS, and Linux.

    ```bash
    dotnet tool install -g microsoft.sqlpackage
    ```

    Verify the installation:

    ```bash
    sqlpackage /version
    ```

    Alternatively, you can download SqlPackage as a zip file from the [SqlPackage download page](https://learn.microsoft.com/sql/tools/sqlpackage/sqlpackage-download).
  </Step>

  <Step title="Install SQL project templates">
    Install the Microsoft.Build.Sql.Templates NuGet package to make the `sqlproj` and `sqlcodeanalysis` templates available to the `dotnet new` command.

    ```bash
    dotnet new install Microsoft.Build.Sql.Templates
    ```
  </Step>

  <Step title="Create a SQL project">
    Create a new SQL project named `ProductsTutorial` using the `sqlproj` template.

    ```bash
    dotnet new sqlproj -n ProductsTutorial
    ```

    This creates a `ProductsTutorial/` folder containing a `ProductsTutorial.sqlproj` file. Any `.sql` files you add to this folder are automatically included in the build.
  </Step>

  <Step title="Add a table">
    Create a file named `dbo.Product.sql` inside the `ProductsTutorial/` folder with the following content:

    ```sql
    CREATE TABLE [dbo].[Product](
        [ProductID] [int] IDENTITY(1,1) NOT NULL PRIMARY KEY,
        [ProductName] [nvarchar](200) NOT NULL
    );
    ```

    <Tip>
      A common convention is to organize files by schema and object type — for example, `Tables/dbo.Product.sql` and `StoredProcedures/dbo.GetProductInfo.sql`. The build process finds all `.sql` files in the project folder and its subfolders automatically.
    </Tip>
  </Step>

  <Step title="Build the project">
    Run `dotnet build` from inside the project folder to compile the `.sql` files into a `.dacpac` artifact.

    ```bash
    dotnet build
    ```

    The output `.dacpac` is written to `bin/Debug/ProductsTutorial.dacpac`.
  </Step>

  <Step title="Add more database objects">
    Extend the project with a stored procedure and a second table. Create `dbo.GetProductInfo.sql`:

    ```sql
    CREATE PROCEDURE [dbo].[GetProductInfo]
        @ProductID INT
    AS
    BEGIN
        SELECT P.ProductID
            , P.ProductName
            , SUM(PO.Quantity) AS TotalQuantity
        FROM dbo.Product P
        LEFT JOIN dbo.ProductOrder PO ON PO.ProductID = P.ProductID
        WHERE P.ProductID = @ProductID
        GROUP BY P.ProductID, P.ProductName
    END
    ```

    The stored procedure references a table that does not exist yet. Create `dbo.ProductOrder.sql` to resolve the dependency:

    ```sql
    CREATE TABLE [dbo].[ProductOrder](
        [ProductOrderID] [int] IDENTITY(1,1) NOT NULL PRIMARY KEY,
        [ProductID] [int] NOT NULL FOREIGN KEY REFERENCES dbo.Product(ProductID),
        [Quantity] [int] NOT NULL
    );
    ```

    Run `dotnet build` again to confirm the project compiles without errors.
  </Step>

  <Step title="Publish to a database">
    Use the SqlPackage `publish` action to deploy the `.dacpac` to a target database. The example below targets an Azure SQL Database using SQL authentication.

    ```bash
    sqlpackage /Action:Publish /SourceFile:"bin/Debug/ProductsTutorial.dacpac" \
        /TargetConnectionString:"Server=tcp:{yourserver}.database.windows.net,1433;Initial Catalog=ProductsTutorial;User ID=sqladmin;Password={your_password};Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;"
    ```

    When the publish command completes, connect to the database to view the created objects. Running the same command again on an updated project will apply only the schema changes — there is no need to drop and recreate the database.
  </Step>
</Steps>
